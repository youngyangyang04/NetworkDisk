# 网盘项目设计模式分析

本文档结合项目代码，说明项目中使用的设计模式及其实现方式。

---

## 1. 策略模式 (Strategy Pattern)

**意图**：定义一系列算法/实现，把它们封装起来，并且使它们可以互相替换，使算法独立于使用它的客户端。

### 1.1 文件存储引擎 (StorageEngine)

**位置**：`networkdisk-common/networkdisk-file`

- **策略接口**：`StorageEngine` 定义存储的通用能力（存文件、删文件、存分片、合并、读文件）。
- **具体策略**：
  - `LocalStorageEngine`：本地磁盘存储（带 `@Primary`，默认注入）
  - `FastDFSStorageEngine`：FastDFS 分布式存储
  - `OssStorageEngine`：阿里云 OSS 存储

**代码实现**：

```java
// 策略接口
public interface StorageEngine {
    void store(StoreFileContext context) throws IOException;
    void delete(DeleteFileContext context) throws IOException;
    void storeChunk(StoreFileChunkContext context) throws IOException;
    void mergeFile(MergeFileContext context) throws IOException;
    void realFile(ReadFileContext context) throws IOException;
}
```

业务层通过依赖注入使用“当前策略”，不关心具体实现：

```java
// UserFileServiceImpl / FileChunkServiceImpl / FileServiceImpl
@Autowired
private StorageEngine storageEngine;
// 调用 storageEngine.store(...)、storageEngine.realFile(...) 等
```

**策略选择**：通过 Spring 注入。默认使用 `LocalStorageEngine`（`@Primary`）；若只启用 OSS 或 FDFS 等实现，可通过条件注解或排除 `@Primary` 切换策略。

---

### 1.2 延迟队列执行器 (DelayQueueExecutor)

**位置**：`networkdisk-common/networkdisk-delay-message`

- **策略接口**：`DelayQueueExecutor<T>`，定义 `queueName()`、`execute(T)`、`take()`。
- **具体策略**：`DeleteShareInfoDelayQueueExecutor`（按 shareId 延迟删除分享）。

**代码实现**：

```java
public interface DelayQueueExecutor<T extends DelayMessage> {
    String queueName();
    void execute(T t);
    T take() throws InterruptedException;
}
```

工厂在容器启动后收集所有 `DelayQueueExecutor` 实现，按 `queueName` 注册并启动后台线程消费：

```java
// DelayQueueExecutorFactory
Map<String, DelayQueueExecutor> delayQueueExecutorMap = 
    event.getApplicationContext().getBeansOfType(DelayQueueExecutor.class);
delayQueueExecutorMap.forEach((k, v) -> {
    executorBeanMap.put(v.queueName(), v);
    this.runExecutor(k, v);  // 为每个队列启动消费线程
});
```

新增一种延迟任务（如过期清理）只需新增一个实现 `DelayQueueExecutor` 的 Bean，无需改工厂逻辑，符合策略+工厂的使用方式。

---

### 1.3 限流器 (RateLimiter)

**位置**：`networkdisk-common/networkdisk-limiter`

- **策略接口**：`RateLimiter`，方法 `tryAcquire(key, limit, windowSize)`。
- **具体策略**：`SlidingWindowRateLimiter`（基于 Redisson 的滑动窗口限流）。

```java
public interface RateLimiter {
    Boolean tryAcquire(String key, int limit, int windowSize);
}
public class SlidingWindowRateLimiter implements RateLimiter { ... }
```

通过配置类注入具体实现，业务代码依赖 `RateLimiter` 接口即可，后续可替换为其他限流算法（如令牌桶）而不影响调用方。

### 1.4 文件类型匹配 (FileTypeEnum + Predicate)

**位置**：`networkdisk-business/networkdisk-files` → `FileTypeEnum`

枚举中为每种文件类型绑定一个 `Predicate<String>`（matcher），按后缀匹配类型，本质是“按类型选择不同匹配策略”：

```java
@Getter
@AllArgsConstructor
public enum FileTypeEnum {
    NORMAL_FILE(1, "NORMAL_FILE", 1, fileSuffix -> true),
    ARCHIVE_FILE(2, "ARCHIVE_FILE", 2, fileSuffix -> 
        Lists.newArrayList(".rar", ".zip", ...).contains(fileSuffix)),
    IMAGE_FILE(7, "IMAGE_FILE", 7, fileSuffix -> 
        Lists.newArrayList(".bmp", ".gif", ".png", ...).contains(fileSuffix)),
    // ...
    private final Predicate<String> matcher;

    public static Integer getFileTypeCode(String fileSuffix) {
        return Arrays.stream(values())
            .sorted(Comparator.comparingInt(FileTypeEnum::getOrder).reversed())
            .filter(value -> value.getMatcher().test(fileSuffix))
            .findFirst()
            .orElseThrow(() -> new RuntimeException("获取文件类型失败"))
            .getCode();
    }
}
```

调用方只传 `fileSuffix`，由枚举通过 `matcher` 策略决定类型，扩展新类型只需加枚举常量。

## 2. 模板方法模式 (Template Method Pattern)

**意图**：在抽象类中定义算法骨架，将某些步骤推迟到子类实现。

**位置**：`networkdisk-common/networkdisk-file` → `AbstractStorageEngine`

抽象类实现 `StorageEngine` 接口，对外方法中完成“校验 + 调用抽象方法”的固定流程，具体存储动作由子类实现：

```java
public abstract class AbstractStorageEngine implements StorageEngine {

    @Override
    public void store(StoreFileContext context) throws IOException {
        checkStoreFileContext(context);  // 1. 通用校验
        doStore(context);                // 2. 子类实现
    }
    protected abstract void doStore(StoreFileContext context) throws IOException;

    @Override
    public void delete(DeleteFileContext context) throws IOException {
        checkDeleteFileContext(context);
        doDelete(context);
    }
    protected abstract void doDelete(DeleteFileContext context) throws IOException;

    // storeChunk、mergeFile、realFile 同理：checkXxx() + doXxx()
}
```

子类（如 `LocalStorageEngine`、`OssStorageEngine`）只实现 `doStore`、`doDelete`、`doStoreChunk`、`doMergeFile`、`doReadFile`，不重复写校验逻辑，符合模板方法模式。

## 3. 门面模式 (Facade Pattern)

**意图**：为子系统提供统一、简化的入口，隐藏内部多服务协作细节。

### 3.1 RPC 门面接口与实现

**位置**：`networkdisk-common/networkdisk-api`（接口）、各 business 模块的 `facade` 包（实现）

- **UserFacadeService**：对“用户查询/注册/修改/认证/激活”等提供统一接口，内部委托 `UserService` 及 Convertor。
- **UserFileFacadeService**：对“创建用户根目录、查询用户文件信息”等提供统一接口，内部委托文件域服务与 Convertor。
- **NoticeFacadeService**：通知相关能力统一入口。

调用方（如 `networkdisk-auth`）只依赖 Facade 接口，不直接依赖多个领域服务和转换逻辑：

```java
// AuthServiceImpl
@Autowired
private UserFacadeService userFacadeService;
@Autowired
private UserFileFacadeService userFileFacadeService;
// 登录/注册时：userFacadeService.query(...)、register(...)、userFileFacadeService.createUserRootFile(...)
```

### 3.2 @Facade 切面（门面横切逻辑）

**位置**：`networkdisk-common/networkdisk-rpc` → `FacadeAspect`、`@Facade`

对标注了 `@Facade` 的方法做统一横切：参数校验、异常转统一响应、日志与耗时统计：

```java
@Around("@annotation(com.disk.rpc.facade.Facade)")
public Object facade(ProceedingJoinPoint pjp) throws Exception {
    // 1. 参数校验 BeanValidator.validateObject(parameter)
    // 2. 执行目标方法 pjp.proceed()
    // 3. 成功时补全 response 的 code/message；异常时构造失败 BaseResponse
    // 4. 统一日志格式（方法名、耗时、success、args、resp、exception）
}
```

门面实现类在对外方法上标注 `@Facade`，即自动获得校验与统一响应格式，无需在每个方法里重复写，可视为“门面 + AOP”的组合使用。

## 4. 工厂模式 (Factory / 注册表式)

**位置**：`networkdisk-common/networkdisk-delay-message` → `DelayQueueExecutorFactory`

- 不手写 `new` 具体执行器，而是从 Spring 容器中取出所有 `DelayQueueExecutor` 类型的 Bean。
- 以 `queueName()` 为 key 注册到 `executorBeanMap`，并为每个执行器启动一条后台消费线程。

```java
@Override
public void onApplicationEvent(ContextRefreshedEvent event) {
    Map<String, DelayQueueExecutor> delayQueueExecutorMap = 
        event.getApplicationContext().getBeansOfType(DelayQueueExecutor.class);
    delayQueueExecutorMap.forEach((k, v) -> {
        executorBeanMap.put(v.queueName(), v);
        this.runExecutor(k, v);
    });
}
```

“产品”是各种 `DelayQueueExecutor` 实现，由 Spring 创建；“工厂”负责发现、注册、启动，相当于**基于容器的工厂 + 注册表**。

---

## 5. 状态机模式 (State Machine)

**位置**：`networkdisk-common/networkdisk-base` → `StateMachine`、`BaseStateMachine`

- **抽象**：`StateMachine<STATE, EVENT>`，仅包含 `transition(state, event)`。
- **实现**：`BaseStateMachine` 用 `Map<String, STATE>` 存 `origin_event -> target` 的转移表，通过 `putTransition(origin, event, target)` 配置，`transition` 时查表得到下一状态，非法转移抛 `BizException`。

```java
public interface StateMachine<STATE, EVENT> {
    STATE transition(STATE state, EVENT event);
}
public class BaseStateMachine<STATE, EVENT> implements StateMachine<STATE, EVENT> {
    private Map<String, STATE> stateTransitions = Maps.newHashMap();
    protected void putTransition(STATE origin, EVENT event, STATE target) {
        stateTransitions.put(Joiner.on("_").join(origin, event), target);
    }
    @Override
    public STATE transition(STATE state, EVENT event) {
        STATE target = stateTransitions.get(Joiner.on("_").join(state, event));
        if (target == null) throw new BizException(...);
        return target;
    }
}
```

当前项目内暂无业务类直接继承或使用该状态机的引用，但基础设施已具备，可用于订单、分享状态、文件状态等流转。

## 6. 转换器模式 (Converter / MapStruct)

**意图**：将一种对象形式转换为另一种（如 DTO ↔ DO、Param ↔ Context），集中映射规则，避免散落的 set/get。

**位置**：各业务模块的 `convertor` 包 + MapStruct

- **UserConvertor**：`UserDO` ↔ `UserInfo`，单例 `INSTANCE = Mappers.getMapper(UserConvertor.class)`。
- **FileConvertor**：多种 Param/VO 与 Context、DO、VO 之间的转换，`componentModel = "spring"` 注入。
- **ShareConvertor**：分享相关 Param → Context、DO → VO。
- **AuthConvertor**：登录/注册 Param → LoginContext/RegisterContext。

**代码示例**（UserConvertor）：

```java
@Mapper(nullValueCheckStrategy = NullValueCheckStrategy.ALWAYS)
public interface UserConvertor {
    UserConvertor INSTANCE = Mappers.getMapper(UserConvertor.class);
    @Mapping(source = "request.id", target = "userId")
    UserInfo mapToVo(UserDO request);
    @Mapping(source = "request.userId", target = "id")
    UserDO mapToEntity(UserInfo request);
}
```

Controller/Facade/Service 中统一通过 Convertor 做入参→Context、DO→VO，保持分层清晰，可视为**转换器/映射器模式**，并由 MapStruct 生成实现。

---

## 7. 上下文/参数对象模式 (Context / Parameter Object)

**意图**：用独立对象封装一次操作所需的全部参数，避免长参数列表，便于扩展和传递。

**位置**：各 domain 的 `context` 包

- 文件域：`CreateFolderContext`、`UploadFileContext`、`FileChunkMergeContext`、`DeleteUserFileContext`、`SaveFileContext`、`FileSearchContext` 等。
- 分享域：`CreateShareContext`、`DeleteShareContext`。
- 认证域：`LoginContext`、`RegisterContext`。

**示例**：

```java
@Getter
@Setter
public class CreateFolderContext implements Serializable {
    private Long parentId;
    private Long userId;
    private String folderName;
}
```

流程为：Controller 接收 Param/VO → 通过 Convertor 转成 Context → Service 只接收 Context，内部再拆字段使用。这样新增参数只需改 Context 和 Convertor，不影响方法签名，符合**参数对象/上下文对象**用法。

---

## 8. 建造者模式 (Builder Pattern)

**位置**：`networkdisk-common/networkdisk-api` → `NoticeResponse.Builder`

用于构建响应对象，链式设置属性并最终 `build()`：

```java
public class NoticeResponse extends BaseResponse {
    public static class Builder {
        private NoticeResponse response;
        public Builder() { response = new NoticeResponse(); }
        public Builder setCode(String code) { response.setResponseCode(code); return this; }
        public Builder setMessage(String message) { ... }
        public Builder setSuccess(boolean success) { ... }
        public NoticeResponse build() { return response; }
    }
}
```

项目中 Builder 使用处较少，但模式清晰，适合复杂响应或可选字段较多的对象构建。

---

## 9. 规格/条件对象模式 (Specification-like / 多态 Condition)

**位置**：`networkdisk-common/networkdisk-api` → `UserQueryCondition` 及其实现

- **抽象**：`UserQueryCondition`（空标记接口，可序列化）。
- **具体条件**：`UserIdQueryCondition`（按 userId）、`UserEmailQueryCondition`（按 email）。

`UserQueryRequest` 持有 `UserQueryCondition`，Facade 层根据具体类型分发到不同查询方式：

```java
// UserFacadeServiceImpl
UserDO userDO = switch (request.getUserQueryCondition()) {
    case UserIdQueryCondition userIdQueryCondition ->
        userService.findById(userIdQueryCondition.getUserId());
    case UserEmailQueryCondition userEmailQueryCondition ->
        userService.findByEmail(userEmailQueryCondition.getEmail());
    default -> throw new UnsupportedOperationException(...);
};
```

调用方通过 `new UserQueryRequest(userId)` 或 `new UserQueryRequest(email)` 构造不同查询条件，服务端通过多态 + switch 选择策略，可视为**条件对象 + 策略**的组合，便于扩展新的查询维度（如按手机号）。

## 10. 适配器模式 (Adapter Pattern)

**位置**：`networkdisk-gateway` → `StpInterfaceImpl`

Sa-Token 要求实现 `StpInterface`，以根据登录 id 返回权限列表和角色列表。项目通过 `StpInterfaceImpl` 适配到现有用户模型（从 Session 取 `UserInfo`，再按 `UserRole`/`UserStateEnum` 返回权限和角色）：

```java
public class StpInterfaceImpl implements StpInterface {
    @Override
    public List<String> getPermissionList(Object loginId, String loginType) {
        UserInfo userInfo = (UserInfo) StpUtil.getSessionByLoginId(loginId).get((String) loginId);
        // 根据 userInfo.getUserRole()、getState() 返回 BASIC/AUTH/FROZEN/NONE 等
    }
    @Override
    public List<String> getRoleList(Object loginId, String loginType) {
        // 根据 userInfo 返回 ADMIN 或 CUSTOMER
    }
}
```

将“现有用户体系”适配成 Sa-Token 需要的接口，即**适配器**。

---

## 11. 观察者/事件监听模式 (Observer / ApplicationListener)

**位置**：`DelayQueueExecutorFactory` 实现 `ApplicationListener<ContextRefreshedEvent>`

在 Spring 容器刷新完成后执行一次“发现所有 DelayQueueExecutor 并启动消费线程”的逻辑：

```java
@Override
public void onApplicationEvent(ContextRefreshedEvent event) {
    Map<String, DelayQueueExecutor> delayQueueExecutorMap = 
        event.getApplicationContext().getBeansOfType(DelayQueueExecutor.class);
    // 注册并 runExecutor(...)
}
```

应用生命周期事件作为“主题”，工厂作为“监听者”，属于**事件驱动/观察者**的典型用法。

---

## 12. 代理模式 + 切面 (AOP / 装饰器思想)

**位置**：多处 `@Aspect` 对注解进行环绕增强

- **FacadeAspect**：对 `@Facade` 方法做参数校验、异常转 BaseResponse、日志与耗时统计。
- **DistributeLockAspect**：对 `@DistributeLock` 方法做加锁/解锁，支持 SpEL 解析锁 key、超时与等待时间。

```java
@Around("@annotation(com.disk.lock.DistributeLock)")
public Object process(ProceedingJoinPoint pjp) throws Exception {
    // 解析 key / keyExpression → 获取 RLock → lock/tryLock → pjp.proceed() → finally unlock
}
```

业务方法只需加注解即可获得“分布式锁”或“门面校验与响应”能力，从调用方看是**透明代理/装饰**，实现上通过 AOP 完成，属于**代理 + 切面**的组合。

---

## 13. 依赖注入与接口编程 (DI + 面向接口)

全项目普遍通过 Spring 注入接口类型（如 `StorageEngine`、`DelayQueueHolder`、`UserFacadeService`、`RateLimiter`），具体实现由配置和条件注解（如 `@Primary`、`@ConditionalOnClass`）决定，便于测试替换和扩展新实现，与上述策略、门面、工厂等模式一起构成可替换、可扩展的架构。

---

## 小结表

| 设计模式         | 主要位置                               | 作用简述                           |
|------------------|----------------------------------------|------------------------------------|
| 策略模式         | StorageEngine、DelayQueueExecutor、RateLimiter、FileTypeEnum | 多实现可替换，按接口编程           |
| 模板方法         | AbstractStorageEngine                  | 固定流程 + 子类实现具体步骤         |
| 门面模式         | *FacadeService + @Facade 切面          | 统一入口、统一校验与响应            |
| 工厂/注册表      | DelayQueueExecutorFactory              | 发现并注册执行器、启动消费          |
| 状态机           | StateMachine / BaseStateMachine        | 状态转移表，可复用于业务状态流转    |
| 转换器           | *Convertor (MapStruct)                 | DTO/DO/Context 间集中映射          |
| 上下文/参数对象  | *Context                               | 封装请求参数，便于扩展与传递        |
| 建造者           | NoticeResponse.Builder                 | 链式构建复杂响应对象                |
| 条件/规格        | UserQueryCondition 及实现              | 多态查询条件，便于扩展查询维度      |
| 适配器           | StpInterfaceImpl                       | 适配 Sa-Token 与现有用户模型        |
| 观察者/事件      | DelayQueueExecutorFactory + ContextRefreshedEvent | 容器就绪后自动注册并启动延迟任务   |
| AOP/代理         | FacadeAspect、DistributeLockAspect     | 统一横切：校验、锁、日志、异常封装  |

以上均基于当前仓库代码归纳，可直接对照类名与包路径在工程中查看具体实现。
