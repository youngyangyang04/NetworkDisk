# 设计模式分析（策略模式与工厂模式）

本文只分析项目里已经落地的实现，不讨论纯概念示例。

## 1. 策略模式（Strategy）

### 1.1 场景与目标

在文件上传、分片、合并、下载等流程中，业务层不希望关心底层存储介质（本地磁盘、OSS、FastDFS）的实现差异。  
因此项目把“文件存储算法”抽象为可替换策略。

### 1.2 代码角色映射

| 策略模式角色 | 项目实现 | 说明 |
|---|---|---|
| Strategy（策略接口） | `StorageEngine` | 定义统一能力：`store / delete / storeChunk / mergeFile / realFile` |
| Context（策略使用方） | `FileChunkServiceImpl`、`FileServiceImpl`、`UserFileServiceImpl` | 只依赖 `StorageEngine`，不感知具体存储实现 |
| ConcreteStrategy（具体策略） | `LocalStorageEngine`、`OssStorageEngine`、`FastDFSStorageEngine` | 各自实现不同存储介质行为 |
| 模板骨架（增强可复用性） | `AbstractStorageEngine` | 统一参数校验与公共流程，子类只实现 `doXxx` |

对应代码位置：
- `networkdisk-common/networkdisk-file/src/main/java/com/disk/file/core/StorageEngine.java`
- `networkdisk-common/networkdisk-file/src/main/java/com/disk/file/core/AbstractStorageEngine.java`
- `networkdisk-common/networkdisk-file/src/main/java/com/disk/file/local/LocalStorageEngine.java`
- `networkdisk-common/networkdisk-file/src/main/java/com/disk/file/oss/OssStorageEngine.java`
- `networkdisk-common/networkdisk-file/src/main/java/com/disk/file/fastdfs/FastDFSStorageEngine.java`
- `networkdisk-business/networkdisk-files/src/main/java/com/disk/files/domain/service/impl/FileChunkServiceImpl.java`
- `networkdisk-business/networkdisk-files/src/main/java/com/disk/files/domain/service/impl/FileServiceImpl.java`
- `networkdisk-business/networkdisk-files/src/main/java/com/disk/files/domain/service/impl/UserFileServiceImpl.java`

### 1.3 运行流程（以“分片上传”为例）

1. 业务服务调用 `storageEngine.storeChunk(...)`。  
2. `AbstractStorageEngine` 先做通用参数校验，再调用 `doStoreChunk(...)`。  
3. 具体策略执行差异化逻辑：  
   - `LocalStorageEngine`：写入本地分片目录。  
   - `OssStorageEngine`：走 OSS 分片上传协议，含 `uploadId` 处理。  
   - `FastDFSStorageEngine`：当前直接抛出“不支持分片上传”。  
4. 业务层拿到统一的 `realPath` 结果继续后续流程。

### 1.4 当前实现特点

- 通过 Spring 注入统一接口，业务代码保持稳定。
- `LocalStorageEngine` 标注了 `@Primary`，默认优先注入本地策略。
- 结合 `AbstractStorageEngine`，本项目是“策略模式 + 模板方法模式”组合实现。

### 1.5 扩展方式

新增一种存储介质时，通常只需要：

1. 新建 `XxxStorageEngine` 继承 `AbstractStorageEngine` 并实现 `doStore/doDelete/doStoreChunk/doMergeFile/doReadFile`。  
2. 注册为 Spring Bean（`@Component`）。  
3. 按需要调整默认注入策略（如 `@Primary` 或 `@Qualifier`）。  

业务层代码通常无需改动。

## 2. 工厂模式（Factory）

### 2.1 场景与目标

延迟队列消费端支持多业务队列。项目需要在启动时：

- 自动发现所有执行器实现；
- 按队列名建立唯一映射；
- 启动对应监听线程并执行分发。

这部分由 `DelayQueueExecutorFactory` 统一完成。

### 2.2 代码角色映射

| 工厂模式角色 | 项目实现 | 说明 |
|---|---|---|
| Product（产品接口） | `DelayQueueExecutor<T extends DelayMessage>` | 统一定义 `queueName / execute / take` |
| ConcreteProduct（具体产品） | `DeleteShareInfoDelayQueueExecutor`（及后续实现） | 各业务队列消费逻辑 |
| Factory（工厂） | `DelayQueueExecutorFactory` | 启动时扫描、注册、启动执行器 |
| Registry（注册表） | `executorBeanMap` | `queueName -> executor` 的唯一映射 |

对应代码位置：
- `networkdisk-common/networkdisk-delay-message/src/main/java/com/disk/delayqueue/executor/DelayQueueExecutor.java`
- `networkdisk-common/networkdisk-delay-message/src/main/java/com/disk/delayqueue/executor/DelayQueueExecutorFactory.java`
- `networkdisk-business/networkdisk-share/src/main/java/com/disk/share/job/delay/DeleteShareInfoDelayQueueExecutor.java`

### 2.3 初始化与分发流程

1. Spring 容器刷新后，`DelayQueueExecutorFactory` 监听到 `ContextRefreshedEvent`。  
2. 通过 `getBeansOfType(DelayQueueExecutor.class)` 收集所有执行器 Bean。  
3. 调用每个执行器的 `queueName()` 做唯一性校验并写入 `executorBeanMap`。  
4. 为每个执行器启动后台监听线程，循环 `take()` 延迟消息。  
5. 消息到达后，投递到 `delayQueueThreadPoolExecutor`，执行 `executor.execute(...)`。  

### 2.4 为什么这里属于“工厂 + 注册表”

- 对象实例本身由 Spring IoC 创建，不是代码里 `new` 出来。  
- 但“执行器的统一发现、装配、生命周期启动”由 `DelayQueueExecutorFactory` 负责。  
- 所以更准确是“容器驱动的工厂模式 + 注册表模式”。

### 2.5 扩展方式

新增一个延迟队列业务时：

1. 新建类实现 `DelayQueueExecutor<T>`。  
2. 实现唯一 `queueName()`、`take()`、`execute()`。  
3. 标注为 Spring Bean（`@Component`）。  

无需改工厂代码，启动后会被自动发现和注册。

## 3. 小结

- 文件模块通过策略模式隔离了存储介质差异，业务层只依赖统一接口。  
- 延迟队列模块通过工厂/注册表在启动阶段完成执行器发现与线程化调度。  
- 两者共同体现了“面向抽象编程 + 开闭原则”的落地方式：新增能力时尽量新增类，而非修改核心流程。
